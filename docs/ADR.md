# 架构决策记录 (ADR) — Hot100 复习工具

## 核心问题
学算法时"看得懂题解,但独立写不出来"——recognition ≠ recall。
本工具全部设计服务于一件事:强制冷启动复现,把"认得出"逼成"写得出"。

> **验证说明**:下方每条决策标注了验证状态。
> `[假设]` = 设计时的判断,尚待真实使用数据支撑;
> `[已验证]` = 有 dogfood 或实测证据。诚实标注未验证项,本身是一种工程态度。

---

## ADR-001 分阶段资源策略(首次 vs 复习)
- **决策**:首次遇到一题 → 静态资源优先(代随链接),AI 不主动讲解;复习卡住 → AI 介入。
- **否决方案**:首次就用 AI 现场讲解。
- **理由**:全新算法,LLM 现编讲解打不过代随的成熟资源;AI 的价值在"针对你的代码纠错",不在"从零讲新知识"。
- **验证**:`[假设]` —— 待填:dogfood 中,首次看代随 vs(假想)AI 讲解,哪个让你更快独立 AC?

## ADR-002 间隔重复用 SM-2,失败即重置
- **决策**:复现打分 1-4 喂入 SM-2;≥3 推进间隔,<3 重置为 1 天。
- **理由**:间隔重复靠跨天再提取造记忆,不是一坐下连刷。
- **验证**:`[假设]` —— 待填:连续 2-3 周后,被 SM-2 安排重复的题,二次冷启动复现率是否上升?给出数字。

## ADR-003 拒绝 while-until-AC 计入评分
- **决策**:一次 session 内反复重做直到 AC,这些重做不计分、不动排期;仅第一次冷启动默写计入 SM-2。
- **否决方案**:grind 到 AC 就算掌握、计入排期。
- **理由**:刚看完提示马上重写是假复现(recognition);连刷与间隔重复是对立哲学,不可缝合。
- **验证**:`[假设]` —— 待填:被告知答案当天"重做 AC"的题,下次冷启动是否仍失败?若失败率高,说明本决策正确。

## ADR-004 判分:LeetCode 定对错,LLM 只判复杂度
- **决策**:正确性由 LeetCode ac/wa 裁定;LLM 仅在 AC 前提下判解法是否最优(2/3/4)。
- **否决方案**:① LLM 读代码直接判对错;② LLM 自构造测试用例心算运行。
- **理由**:LLM 靠读代码判正确性会放水;正确性客观可测,外包给判题机,LLM 只做擅长的静态复杂度分析。
- **关键洞察**:暴力解 AC ≠ 掌握,记 <3、照样重置。
- **验证**:[已验证]:三个真实案例——①二分查找:LLM 把 WA 代码判为通过(假阳性);②螺旋矩阵 & ③删链表倒数N:LLM 对两段 AC 的最优解编造了不存在的 bug 并标注行号(假阴性)。结论:LLM 靠读代码判正确性,既会放过错的、也会冤枉对的,正确性必须且只能由判题机裁定。

## ADR-005 不做三层 Multi-Agent 架构
- **决策**:不做 router → subgraph → tools 意图路由。
- **否决方案**:模仿成熟项目的 L1 路由 / L2 子图 / L3 原子工具。
- **理由**:路由为"单入口自动分发多意图"而生;本工具是单人驱动,用户自己点函数,不存在需路由的问题。为不存在的问题做工程 = 过度设计。
- **验证**:`[已验证]` —— 设计层面成立:实际使用中从未出现"不知道该调哪个流程"的情况,路由无用武之地。

## ADR-006 hint 默认 direct,而非苏格拉底
- **决策**:提示默认直接指出错误,苏格拉底式降为 opt-in。
- **理由**:用户求提示时通常已耗时很久,层层反问是折磨;"直接讲一次"安全——SM-2 保证此题会冷启动再考。
- **验证**:`[假设]` —— 待填:dogfood 中你实际用 direct 还是 socratic?用脚投票的结果即证据。

## ADR-007 向量库的职责是个性化,不是找 bug
- **决策**:RAG 召回用于"把这次错接到用户自己的 EP/KP",不承担识别错误。
- **理由**:识别 bug 是 LLM 能力(与库大小无关);库的价值随用户积累错误模式而增长,是记忆/个性化层。
- **验证**:`[假设]` —— 待填:库扩大后,带 EP/KP 引用的提示是否比纯 LLM 提示更有用?(需库长大后做小型消融)

## ADR-008 可推导的状态不持久化
- **决策**:刷题进度不建表——"下一题"= 第一道无 review_state 记录的题(按 study_order)。
- **否决方案**:单独建刷题顺序表 / 进度表。
- **理由**:顺序隐含在 study_order,进度隐含在 review_state 的有无;存可推导状态会产生两份真相,迟早不一致。
- **验证**:`[已验证]` —— 设计层面成立,且呼应一次实际 bug:run_review_session 曾存两份导致行为分叉。
